Einleitung

Die nntpmlib ist eine C-Bibliothek, die einen Teil des nntp-Protokoll und der common nntp-extensions implementiert. Um keinen großen overhead zu erzeugen, sind die einzelnen Funktionen sehr nah am Protokoll selbst: Die meisten Funktionen entsprechen genau einem Befehl des nntp-Protokolls. Außerdem gibt es einige zusätzliche Funktionen zum Parsen der Antworten und für ähnliche Aufgaben.

Das nntp-Protokoll

Das nntp-Protokoll ist ein rein textbasiertes Netzwerk-Protokoll, welches auf TCP aufsetzt. NNTP-Server laufen üblicherweise auf Port 119. Die genaue Liste der NNTP-Befehle entnimmt man am besten dem RFC977. Folgende Befehle existieren laut RFC:

  • article [MessageID|Number]
  • body [MessageID|Number]
  • group newsgroup
  • head [MessageID|Number]
  • help
  • ihave
  • last
  • list [active|newsgroups] [group_pattern]
  • newgroups yymmdd hhmmss [GMT] [<distributions>]
  • newnews newsgroups date time [GMT] [<distribution>]
  • next
  • post
  • quit
  • slave
  • stat [MessageID|Number]

Da sich dieser RFC in der Praxis als unzureichend herausgestellt hat, gibt es im RFC2980 sogenannte "common NNTP-Extensions". Nach diesem RFC existieren zahlreiche weitere Befehle, die ich hier nicht alle aufzähle. Unter diesen ist in dieser Bibliothek ausschließlich der XOVER-Befehl, der dazu dient, auf bequeme Weise Artikel-Listen zu holen, implementiert.

Es gibt fast keine News-Server, die alle Befehle unterstützen, und selbst die implementierten Befehle sind bei vielen Servern unvollständig oder abweichend implementiert, d.h. man kann einige Optionen nicht benutzen und muß mit ungewöhnlichen Antwortcodes rechnen. Des weiteren implementieren viele News-Server eigene Befehle, welche man durch Ausführen des Kommandos help erfährt.

Auch diese Bibliothek implementiert nur einen winzigen Teil der möglichen Befehle. Und zwar habe ich diejenigen geschrieben, die für mich persönlich am wichtigsten waren, und die ich für die am häufigsten implementierten halte. Natürlich ist die Auswahl rein willkürlich. Bei Bedarf kann man im Abschnitt über die Implementation eigener Kommandos erfahren, wie man diese Bibliothek am besten erweitert.

Um einige der folgenden Funktionen zu verstehen, muß man noch wissen, daß ein NNTP-Server für jede Verbindung, die ein Client zu ihm aufbaut, Zeiger auf eine "aktuelle Gruppe" und einen "aktuellen Artikel" verwaltet. Ruft man beispielsweise den Befehl article ohne Argumente auf, so wird der "aktuelle Artikel" geschickt. Auch die Befehle next, last und group dienen dazu, diese Zeiger zu verändern.

Die library führt im Augenblick noch kein Buch über diese beiden Zeiger. Dies wird der Anwendung selber überlassen.

Weiterhin muß man noch wissen, daß jeder Artikel im usenet eine eindeutige Message-ID hat. Innerhalb jeder Gruppe hat jeder Artikel vom server auch noch eine Artikel-Nummer bekommen. Während die Message-ID globale Gültigkeit besitzt, hat die Nummer nur Bedeutung innerhalb einer Gruppe und eines servers. Meistens numerieren die Server die ankommenden Artikel einfach monoton steigend durch. Ältere Artikel werden irgendwann gelöscht.

Grundsätzliches zur Benutzung der Bibliothek

Logging und Fehlermeldungen

Vielen Funktionen der library werden zwei Funktionszeiger

   void (*log_func)(const char*),
   void (*error_func)(const char*),

übergeben. Ist die log_func verschieden von NULL, so werden alle Status-Ausgaben des servers an diese Funktion übergeben. Ist die error_func verschieden von NULL, so wird, falls ein Fehler auftritt, eine Fehlermeldung an diese Funktion übergeben.

Diese beiden Funktionen sollen ausschließlich der Ausgabe dienen. Bei fast allen Funktionen kann man zuzsätzlich am Rückgabwert erkennen, ob sie erfolgreich waren oder nicht. Bei den Funktionen, die einen Integer-Wert zurückgeben, bedeutet, sofern nicht anders vermerkt, eine 0, daß der Befehl erfolgreich ausgeführt werden konnte, eine andere Zahl, daß ein Fehler auftrat.

Will man die Fehlermeldungen weiterverwenden, muß man sie kopieren, da der durch die Zeichenketten belegte Speicherplatz nach Aufruf dieser Funktionen wieder freigegeben wird.

Herstellen und Schließen einer Verbindung

Hierzu gibt es die beiden Funktionen

   nntp_server *nntp_connect (const char *servername, 
                                    unsigned short int port,
                                    void (*log_func)(const char*),
                                    void (*error_func)(const char*),
                                    int *connection_error);
   void nntpserver_close (nntp_server *ns,
                          void (*log_func)(const char*),
                          void (*error_func)(const char*));

Der Typ nntp_server ist dabei wie folgt deklariert:

   typedef struct {
     FILE *fin, *fout;
   } nntp_server;

Die beiden Elemente dienen dem Einlesen vom socket bzw. der Ausgabe auf den socket.

Die Funktion nntp_connect versucht eine TCP-Verbindung zum in servername angegebenen server auf den angegebenen Port herzustellen. Der servername kann eine IP-Adresse oder ein Name sein. Im Erfolgsfall liefert die Funktion einen Zeiger auf ein neu allokiertes Element vom Typ nntp_server zurück. Ansonsten einen Null-Zeiger.

Tritt ein Fehler beim Aufbau der TCP-Verbindung auf, so wird (falls !=NULL) connection_error gesetzt. Lehnt der server die Verbindung ab, so wird connection_error auf 0 gesetzt.

Die Funktion nntpserver_close schließt die Verbindung und gibt den durch den nntp_server* reservierten Speicherbereich wieder frei.

Auslesen von Rückgabezeilen

Einige Funktionen veranlassen den Server dazu, viele Text-Zeilen zurückzugeben. Diese werden von den Funktionen selber nicht ausgelesen, sondern die Anwendung muß dies selber machen. Hierzu dienen die Funktionen

   char *get_line_from_server (nntp_server *ns, int *fin);
   char **get_lines_from_server (nntp_server *ns, int approx_lines, int *no_lines);

Die erste Funktion liest eine Zeile vom Server und gibt sie in einer neu allokierten Zeichenkette zurück. Die LF- und CR-Zeichen, mit denen jede vom server gelieferte Zeile endet, werden von der Bibliothek entfernt. Ist keine weitere Zeile mehr vorhanden, gibt die Funktion NULL zurück und setzt, sofern fin!=NULL ist, *fin=1. Ansonsten ist *fin==0.

Tritt beim Lesen ein Fehler auf, so liefert die Funktion ebenfall NULL, aber *fin ist weiterhin 0. (An der Integer-Zahl *fin kann man also, sofern sie existiert, erkennen, ob ein Lesefehler aufgetreten ist, oder ob der Lesevorgang regulär beendet wurde. Da man in diesem Fall aber sowie nichts mehr machen kann, reicht es vermutlich meistens zu lesen, bis die Funktion NULL zurückgibt.)

Die zweite Funktion liest solange Zeilen vom server, bis die Übertragung beendet ist. Sie speichert alles in einem nullterminierten array von neu allokierten char* ab, der zurückgegeben wird. In approx_lines muß die erwartete Anzahl der Zeilen angegeben werden. Falls dieser Platz nicht reicht, wird der allokierte Speicher jedesmal um 50% vergrößert. Ist approx_lines==0, so wird auf einen default-Wert gesetzt. In no_lines wird (falls !=NULL) die Anzahl der ausgelesenen Zeilen zurückgegeben.

Gruppen

Der Dateityp für Newsgroups

Um eine Gruppe mit eventuellen Zusatzinformationen abspeichern zu können, gibt es folgenden Dateityp:

   typedef struct {
     char *name;
     long int first, last;
     int flags;
   } group_t;

name ist der Name der Gruppe, first und last sind die Nummer des ersten und des letzten Artikels in dieser Gruppe auf diesem Server. Es muß allerdings nicht zu jeder Nummer zwischen diesen beiden auch einen Artikel auf dem Server geben. flags gibt es bisher nur eines:

   #define POSTING_ALLOWED 1

Für die Verwaltung von Gruppen gibt es folgende Hilfsfunktionen:

   group_t *new_group_t ();
   void free_group_t (group_t *g);
   void printf_group_t (group_t *g);

Die erste Funktion allokiert Speicherplatz und initialisiert ihn mit Nullen. Die zweite Funktion gibt den Speicherplatz (einschließlich des durch g->name allokierten Platzes) wieder frei.

Die dritte Funktion gibt die zu einer Gruppe gehörenden Informationen auf der Standard-Ausgabe aus. Sie diente eigentlich nur zum debuggen.

Gruppen-Listen anfordern

Wenn man eine Verbindung zu einem server hergestellt hat, will man wissen, welche Gruppen dieser führt:

   int nntp_list_groups (nntp_server *ns,
                         void (*log_func)(char*),
                         void (*error_func)(char*))
   int nntp_newgroups (nntp_server *ns,
	               const char *datetime_string, int gmt,
                       const char *distribution,
                       void (*log_func)(char*),
                       void (*error_func)(char*))
   group_t *parse_list_line (char *line)

Die erste Funktion setzt einen LIST-Befehl an den Server ab. Dieser sollte daraufhin alle seine Gruppen schicken, jede in einer eigenen Zeile, versehen mit einigen Zusatzinformationen, die man mit Hilfe der letzten Funktion parsen kann. Die einzelnen Zeilen müssen wie erwähnt mit get_line_from_server ausgelesen werden.

Mit Hilfe der Funktion nntp_newgroups kann man, sofern der Befehl vom Server richtig implementiert ist, alle Gruppen aus bestimmten Hierarchien anfordern, die neuer als datetime_string sind. Dieser muß das Format "YYMMDD hhmmss" (ohne die Anführungszeichen) haben. Es wird angenommen, daß es sich um die Zeit der Servers handelt, außer, wenn gmt!=0 ist, in welchem Fall angenommen wird, daß es sich um eine GMT-Zeit handelt.

Das optionale Argument distribution hingegen darf auch NULL sein. Andernfalls muß es einen string wie beispielsweise "<alt.fan,de,comp>" enthalten (ohne die Anführungszeichen): Innerhalb spitzer Klammern eine durch Kommas getrennte Liste von Hauptgruppen, in denen nach neuen Gruppen gesucht werden soll. (Hier werden alle neuen de, comp und alt.fan-Gruppen gesucht.)

Die letzte Funktion nimmt eine Zeile, wie sie nach einem der beiden obigen Befehle vom Server zurückgegeben wird, parst sie, und schreibt das Ergebnis in eine neu allokierte Variable vom Typ group_t*.

Aktuelle Gruppe setzen

Die aktuelle Gruppe kann mit mit der Funktion

   int nntp_group (nntp_server *ns, const char *group,
                   int *n, int *f, int *l,
                   void (*log_func)(char*),
                   void (*error_func)(char*));

gesetzt werden. In den Variablen *n, *f und *l werden die ungefähre Anzahl der Artikel in dieser Gruppe, die Nummer des ersten Artikels, und die Nummer des letzten Artikels abgespeichert. Braucht man diese Informationen nicht, kann aber auch NULL übergeben werden.

Artikel

Der Datentyp für Artikel-Informationen

Um die wichtigsten Informationen über einen Artikel festzuhalten gibt es den Datentyp article_t:

   typedef struct {
     char *subject;
     char *from;
     char *id;
     char *newsgroups;
     char *date;
     char *references;
     char *fu2;
     char *reply_to;
     int lines;
     long no;
     long bytes;
   } article_t;

Wichtig im folgenden sind vor allem die Message-ID id, welche stets in spitzen Klammern eingefaßt sein muß. (Sie hat beispielsweise die Form <983hj$it$bbmb@news.irgendwo.de>.) Die Artikel-Nummer ist in no abgespeichert.

Zum Anlegen, Löschen und Kopieren von Variablen dieses Typs dienen die Funktionen

   article_t *new_article_t ();
   void free_article_t (article_t *a);
   article_t *copy_article_t (const article_t *a);
   void printf_article_t (article_t *a);

new_article_t allokiert Speicherplatz für einen neuen article_t, setzt alle Werte auf 0, und gibt einen Zeiger darauf zurück. free_article_t versucht den Speicherplatz aller char*, die verschieden von NULL sind, wieder freizugeben, und gibt anschließend den durch die Struktur selber belegten Speicherplatz wieder frei. copy_article_t allokiert Speicher für einen neuen Artikel und kopiert die Daten des übergebenen dort hinein. Auch für alle Zeichenketten wird mittels strdup neuer Speicherplatz allokiert.

printf_article_t gibt die für einen Artikel verfügbaren Informationen auf der Standardausgabe aus. Sie diente mir im wesentlichen zum debuggen.

Zum Hinzufügen von Informationen über einen Artikel zu einer article_t-Struktur gibt es die Funktionen

   int insert_value_into_article_t (article_t *a, 
                                    const char *name, const char *value);
   int parse_article_header_line (article_t *a, const char *line);

Die erste Funktion ist, denke ich, selbsterklärend. Der Aufruf erfolgt beispielsweise mit

   insert_value_into_article_t (a, "from:", "myname@irgendwo.de");

In diesen Fall wird eine Kopie der zweiten Zeichenkette an der betreffenden Stelle eingefügt. Groß- und Kleinschreibung im Parameter name ist irrelevant.

An die zweite Funktion wird eine Zeile, so wie sie der Server dem Client geschickt hat, übergeben, und sie versucht, für den Artikel relevante Informationen aus ihr herauszubekommen und einzutragen.

Beide Funktionen geben 0 zurück, falls die Information hinzugefügt werden konnte, und -1 andernfalls. Das letztere tritt beispielsweise auf, wenn die in der Kopf-Zeile oder in name und value übergebene Information gar nicht im article_t-Typ gespeichert wird.

Will man den article_t erweitern, so reicht es, diese Struktur zu erweitern und die Funktionen oben anzupassen. (Dies sollte aber kein Problem sein.) Ich habe die Anzahl der Daten aus Performance-Gründen eher klein gehalten.

Anfordern einer Artikel-Liste

Um einen Überblick über alle in der aktuellen Gruppe vorhandenen Artikel zu bekommen, gibt es die Funktionen

   int nntp_xover (nntp_server *ns, int min_no, int max_no,
                   void (*log_func)(char*),
                   void (*error_func)(char*));
   article_t *parse_xover_line (char *line, const char **format);

Die erste Funktion schickt einen XOVER-Befehl an den Server: Jede Rückgabezeile des Servers enthält Informationen über einen Artikel. Mit Hilfe der Parameter min_no und max_no kann man einen Artikel-Nummern-Bereich angeben. Im Detail haben diese Zahlen dabei folgende Bedeutung:

min_nomax_no
-1 -1es wird kein Bereich angegeben
min_no==max_no>=0nur diese Nummer wird angegeben
>=0-1alles ab min_no
>=0>=0von min_no bis max_no

Die Funktion parse_xover_line dient zum parsen einer Rückgabezeile, wie sie der Server nach einem solchen XOVER-Befehl liefert. Das Ergebnis wird in einem neu allokierten article_t* zurückgegeben. format ist ein nullterminierter array von Zeichenketten, welcher angibt, in welchem Format der Server antwortet. Wirt hier NULL übergeben, wird als default folgendes genommen:

   "subject:", "from:", "date:", "message-id:", "references:", "bytes:", "lines:"

Ein Server darf noch mehr Informationen liefern. In diesem Fall kann man das Format mit Hilfe des (nicht implementierten) Befehls LIST OVERVIEW.FMT anfordern.

Nächster und vorheriger Artikel

   int nntp_next (nntp_server *ns, int *n, char **a, int mode,
                  void (*log_func)(char*),
                  void (*error_func)(char*));

dient dazu, den Zeiger "aktueller Artikel" auf dem server auf den nächsten bzw. auf den vorherigen Artikel zu setzen, je nachdem, welchen der beiden Werte

   SET_TO_NEXT
   SET_TO_LAST

der Parameter mode hat. Im Erfolgsfall wird in n wird (falls !=NULL) die Nummer des Artikels zurückgegeben, und in a (falls !=NULL) die Artikel-Id (neu allokiert).

Anfordern von Artikeln

Nun kommen wir zu einer ganz zentralen Funktion. Derjenigen nämlich, die einen Artikel tatsächlich vom server anfordert:

   int nntp_get_article (nntp_server *ns, article_t *ar,
                         int by_mode, int get_what,
                         void (*log_func)(char*),
                         void (*error_func)(char*));

Diese Befehl fordert einen bestimmten Artikel vom Server an. War die Anfrage erfolgreich, muß die Anwendung den Artikel dann selber mit Hilfe der get_line_from_server-Funktion auslesen.

Der Parameter by_mode muß einen der Werte

   BY_MESSAGEID
   BY_MESSAGENO
   BY_NONE

haben und gibt an, ob der Artikel mit der in ar gesetzten Message-ID, der Artikel mit der in ar gesetzten Artikel-Nummer, oder der "aktuelle Artikel" geladen werden soll. Im letzten Fall darf ar auch NULL sein. Der Übergabeparameter ar bleibt in jedem Fall unverändert.

Der Parameter get_what muß einen der Werte

   GET_HEAD
   GET_BODY
   GET_ALL
   GET_STAT

haben. Er gibt an, welcher Teil des Artikels geliefert werden soll: Nur der Artikel-Kopf, nur der Artikel-Körper (ohne Kopf), beide (in diesem Fall durch eine Leerzeile voneinander getrennt), oder gar nichts. Dieser letzten Fall GET_STAT kann beispielsweise dafür verwendet werden, zu testen, ob es einen bestimmten Artikel gibt. Im Falle, daß by_mode==BY_MESSAGENO ist, wird, soweit ich weiß, vom Server der Zeiger "aktueller Artikel" auf den entsprechenden Artikel gesetzt.

Parsen von Artikel-Köpfen

Hat man mit obiger Funktion get_article Den Kopf eines Artikels oder einen ganzen Artikel empfangen, so kann mit mit der Funktion

   article_t *nntp_read_header_lines (nntp_server *ns);

die Kopfzeilen nach relevanten Informationen parsen. Diese werden in einer neu allokierten article_t-Variablen abgespeichert, und es wird ein pointer auf diese zurückgegeben. Da der Artikel-Kopf die Artikel-Nummer nicht enthält (kann er ja nicht), kann das Ergebnis diese in keinem Fall enthalten. Man muß sie, falls erwünscht, nachträglich per Hand setzen.

Da nicht alle in der Struktur article_t vorgesehenen Einträge obligatorisch in einem Artikel-Kopf sind, muß man damit rechnen, daß einige char* den NULL-Wert haben.

Die Funktion liest die Antwortzeilen des Servers durch, parst sie, und verwirft sie danach. Im Falle, daß der gesamte Artikel angefordert wurde, hört die Funktion bei einer Leerzeile auf zu lesen. Den eigentlichen Artikel muß man danach per Hand auslesen. Im Falle, daß nur der Artikel-Kopf angefordert wurde, liest sie alles aus.

Posten von Artikeln

   int nntp_post (nntp_server *ns, const char **text,
                  void (*log_func)(char*),
                  void (*error_func)(char*));

Mit Hilfe dieser Funktion kann man einen Artikel zum Server posten. text muß einen nullterminierten array mit dem zu postenden Text enthalten (Jedes test[i] eine Zeile oder auch nicht.) Der Anwender muß selber dafür Sorge tragen, daß der Text nach RFC0850 wohlformatiert ist. Das bedeutet insbesondere:

  • keine Sonderzeichen
  • Jede Zeile endet mit einem <CR_LF><CR_LF> (d.h. "\r\n").
  • Am Anfang des Artikels kommt, durch eine Leerzeile vom Rest abgetrennt, der Artikel-Kopf. (Modernere Server verlangen nur noch From, Subject und Newsgroups und können den Rest selber ausfüllen.)

Die abschließende Zeile, dir nur aus einem Punkt besteht, schickt die Funktion schließlich selber ab. Hat das Posten geklappt, gibt sie 0 zurück. Gibt sie -1 zurück, konnte das Posting gar nicht abgesetzt werden. Vermutlich erlaubt der Server dies nicht. Gibt sie -2 zurück, hat sie zwar versucht, das Posting abzusetzen, aber der Server hat es, wahrscheinlich aufgrund eines Formatierungsfehlers, oder weil das Posten in die angegebene Gruppe verboten ist, zurückgewiesen.

Sonstige Kommandos

Anfordern der vom Server unterstützten Befehle

Die vom Server unterstützten Befehle kann man mit Hilfe der Funktion

   int nntp_help (nntp_server *ns, 
                  void (*log_func)(char*), void (*error_func)(char*));

anfordern. Der Server wird als Antwort eine Liste seiner Kommandos samt Syntax schicken, und zwar jedes Kommando in einer Zeile. Diese Zeilen müssen vom Anwender selber ausgelesen und interpretiert werden. (Beispielsweise mit Hilfe der get_line_from_server-Funktion.

Setzen auf News-Reader- oder Slave-Mode

Einige News-Server verhalten sich unterschiedlich, je nachdem, ob der Client ein News-Reader oder ein anderer News-Server ist. Beispielsweise kann einem anderen News-Server höhere Priorität gegeben werden, weil er mehrere Personen bedient. Oder es können für einem News-Reader performantere Algorithmen zum suchen von einzelnen Artikeln durchgeführt werden.

Um dem Server mitzuteilen, ob wir ein News-Reader oder ein News-Server sind, gibt es die Funktion

   int nntp_set_mode_reader (nntp_server *ns, int mode, int *posting_allowed,
	                     void (*log_func)(char*), void (*error_func)(char*));

Die Variable mode muß dabei einen der Werte NNTP_MODE_READER (wir sind ein News-Reader) oder NNTP_MODE_SLAVE (wir sind ein News-Server) haben.

Im Falle mode==NNTP_MODE_READER wird in der Variable posting_allowd, sofern sie !=NULL ist, eine 0 oder ein 1 zurückgegeben, je nachdem, ob Posten erlaubt oder verboten ist.

Dieser Befehl sollte in der Regel keinen Einfluß auf das Funktionieren der Verbindug haben. Bei großen Gruppen beschleunigt er die Arbeit manchmal ein wenig. Wenn ein Server auf disen Befehl einen Fehlercode zurückliefert, z.B. weil er ihn nicht versteht, ist er sowie so obsolet.

Implementation eigener Kommandos

Da für das Schreiben auf und das Lesen vom Socket Variablen vom Typ FILE* definiert sind, ist das eigene Implementieren von zusätzlichen Kommandos ziemlich einfach: Mit fprintf kann man auf den server schreiben (nicht vergessen, danach einen fflush zu machen!), mit fgets oder ähnlichen Befehlen kann man vom server lesen. Mit den folgenden Funktionen kann man sich allerdings einige Arbeit sparen.

Senden von Kommandos

Viel Arbeit nimmt einem die folgende Funktion ab:

   int nntp_send_command (nntp_server *ns, char *command,
                          char **response, int *rcode,
                          int *ok_codes, int *err_codes,
                          void (*log_func)(char*),
                          void (*error_func)(char*));

Diese Funktion schickt an den Server das Kommando command. In dieser Zeichenkette steht also der NNTP-Befehl samt sämtlicher Optionen oder Parameter. Das Kommando darf nicht mit \r\n enden, da die Funktion selber diese Zeichen hinten dranhängt.

Der Server wird nun eine Antwortzeile schicken, die mit einem numerischen Antwortcode beginnt. Ist response!=NULL, so wird in *response (neu allokiert) die Antwortzeile zurückgegeben. ist rcode!=NULL, so wird in *rcode der numerische Antwortcode zurückgegeben.

ok_codes ist ein nullterminierter Array, der diejenigen Antwortcodes enthält, die als Erfolg gewertet werden sollen. Entsprechend ist err_codes ein nullterminierter Array, er die Antwortcodes enthält, die der Server bei einem Fehlschlagen schicken darf. Die allgemeinen Antwortcodes 400 und >=500 sollten hierin nicht enthalten sein, da sie unabhängig abgefragt werden. Im Falle, daß der Antwortcode nicht in ok_codes ist, wird (sofern !=NULL) die error_func aufgerufen. Ist die Antwort weder eine allgemeine Antwort, nicht in ok_codes noch in err_codes wird zusätzlich eine Zeile wie "illegale Antwort vom Server" nach error_func ausgegeben.

Eventuell liefert der Server nach dieser ersten Antwortzeile noch weitere Datenzeilen, die der Anwender dann per Hand auslesen muß.

Hier ein kleines Beispiel: Eine mögliche Implemtation des NEXT-Kommandos:

  int  ok_codes[]={223, 0};
  int err_codes[]={412, 420, 421, 422, 0};
  int err;
  char *response;

  err = 
    nntp_send_command ("next", &response, NULL, ok_codes, err_codes, NULL, NULL);

(Auf eine Fehlerbehandlung, das Auslesen des Antwortcodes und das Parsen der Antwortzeile wurde hier verzichtet.)

Auslesen einer Server-Antwort

Normalerweise liest die Funktion nntp_send_command die Antwort des Servers bereits selber aus. Sollte es einmal nötig sein, dieses selber zu tun, beispielsweise weil der Server, wie nach den POST-Befehl zwei Antwortzeilen zurückschickt, hilft einem dabei die Funktion

  int read_server_response (nntp_server *ns, char **response, int *rcode,
                            int *ok_codes, int *err_codes,
                            void (*log_func)(char*),
                            void (*error_func)(char*));

Die Bedeutung der Übergabe- und Rückgabe-Parameter ist hierbei genau wie bei nntp_send_command. Tatsächlich ist es so, daß diese ihre Parameter einfach nur an read_server_response weitergibt.
