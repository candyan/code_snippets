Einführung

ploglib.c ist eine Bibliothek, die dazu dienen soll, Log-Dateien von HTTP-Servern zu parsen. Da dies mit Hilfe von regulären Ausdrücken geschieht, sollte man sich mit diesen auskennen, wenn man die Bibliothek benutzen will.

Um eine Log-Datei zu parsen, muß man der Bibliothek zunächst das vom Server verwendet Log-Format in Form eines abgewandelten regulaeren Ausdrucks mitteilen. Dieser wird dann in eine für C-Programme auswertbare Form umgewandelt, und man kann anfangen, die einzelnen Zeilen zu parsen. Der Parser gibt dann Variablen vom Typ access_t zurück, in denen alle gewünschten Daten sauber aufgeschlüsselt stehen.

Log-Formate

Der Log-Format-Typ

Log-Formate werden in der Bibliothek durch einen Typ

   logformat_t

repräsentiert. Zum Anlegen und freigeben von Variablen dieses Typs gibt es die Funktionen

   logformat_t *new_logformat (const char *formatstring);
   logformat_t *copy_logformat_t (logformat_t *lf);
   void free_logformat_t (logformat_t *lf);

Wie der bei der Anlage einer Variable mit new_logformat zu übergebene formatstring auszusehen hat, wird im nächsten Abschnitt besprochen.

Die Funktion

   int access_t_equal (access_t *a1, access_t *a2)

vergleicht zwei Variablen von diesem Typ. Sie gibt, falls diese gleich sind, -1 zurück, falls sie sich in irgendeinem Punkt unterscheiden, 0.

Der Format-String

Der vom Benutzer übergebene formatstring ist ein regulärer Ausdruck, der allerdings zusätzlich folgende Elemente enthalten darf:

Abk.NameBedeutung
%S requester Die IP-Adresse des anfragenden Clients in der Form x.x.x.x (x=0...255).
%C command Das Kommando, das der Client an den Server geschickt hat. (Beispielsweise GET oder HEAD).
%c return code Der Antwort-Code des Servers (z.B. 200 für "ok" oder 404 für "file not found").
%r request Die vom Client angeforderte Datei.
%b bytes Anzahl der vom Server zurückgelieferten Bytes.
%d domain Die Domain (genauer: der vom Client angegebene Hostname), von der die Datei angefordert wird. Dies kann dann wichtig sein, wenn ein Server verschiedene Domains bearbeitet.
%f referrer Falls der Client einem Link gefolgt ist, steht die Ursprungsseite hier drin.
%B browser Wie der vom Client benutzte Browser sich identifiziert.
%Y year vierziffrige Jahresangabe.
%M month(numeric) Monat, 1- oder 2-ziffrig.
%N month Monat, dreibuchstabige Abkürzung.
%D day(numeric) Tag, 1- oder 2-ziffrig.
%E day Tag, dreibuchstabige Abkürzung.
%h hour Stunde, 1- oder 2-ziffrig.
%m minute Minute, 1- oder 2-ziffrig.
%s second Sekunde, 1- oder 2-ziffrig.

Hier mal ein (zugegebenermaßen konstruiertes) Beispiel. Die Zeilen in den Log-Dateien des Servers sollen wie folgt aussehen:

   61.103.208.80 - [18/Jun/2001:00:08:00] "GET /test/index.html HTTP/1.0" 200 989

Die an die Funktion new_logformat übergebene Zeichenkette, mit der man alle in dieser Zeilen enthaltenen Informationen extrahieren kann, lautet

   ^%S - \[%D/%N/%Y:%h:%m:%s\] "%C %r .+" %c %b$

Intern geschieht dabei folgendes: Die Funktion ersetzt die Zeichenketten %S, %D usw. durch reguläre Ausdrücke wie (.*) oder (.[0-9]). Weiterhin merkt sie sich, an welcher Stelle der Zeichenkette welche Informationen stehen und speichert dies alles in der Variablen vom Typ logformat_t ab. Der Benutzer braucht sich um diese Sachen nicht weiter kümmern. Das einzige, was er wissen sollte, ist, daß die übergebene Zeichenkette keine Teil-Ausdrücke in runden Klammern enthalten darf, die bei regulären Ausdrücken Substrings herauslösen.

Monats- und Tagesabkürzungen

Wie man obiger Tabelle entnimmt, erkennt die Bibliothek dreibuchstabige Monats- und Tages-Abkürzungen. Man beachte, daß dies nicht die Abkürzungen aus der locale des Benutzers sein werden, sondern aus der locale der uid des Servers, von dem die Log-Dateien stammen. Dies dürfte fast immer die C-locale sein, und die Abkürzungen die englischen Standard-Abkürzungen. Diese werden auch gesetzt, wenn man mit new_logformat ein neues Log-Format anlegt.

Dennoch hat der Benutzer die Möglichkeiten, die für ein Log-Format benutzten Abkürzungen manuell zu ändern:

   void change_month_abbrevs (logformat_t *lf, char **new_months);
   void change_days_abbrevs (logformat_t *lf, char **new_days);

Übergeben wird ein array aus 12 bzw. 7 Zeichenketten. Die Abkürzungen für die Tage beginnen dabei mit 0 für Sonntag. Die Monatsabkürzungen mit 0 für Januar. Die übergebenen Werte werden in einen neuen Speicherbereich kopiert. Die Variablen new_month und new_days können, falls gewünscht, also nach dem Aufruf wieder freigegeben werden.

Hat man eine dreibuchstabige Monats- oder Tages-Abkürzung, so kann man die zugehörige Nummer mittel der Funktionen

   int find_day3_abbrev (logformat_t *lf, char *day);
   int find_mon3_abbrev (logformat_t *lf, char *month);

feststellen. Wie eben ist der Tag 0 der Sonntag. Der Januar ist der Monat Nr.1, der Dezember der Monat Nr.12.

Der Default-Filename

Auf den meisten Servern gibt es einen Default-Dateinamen, meist index.html, der ausgeliefert wird, wenn ein Client ein Verzeichnis als URL angibt. Man hat die Möglichkeit, der Bibliothek mitzuteilen, daß sie an requests oder referrer, die mit einem slash '/' enden, diesen Dateinamen dranhängen soll. Hierzu muß man zwei Dinge tun:

Erstens muß man den Default-Dateinamen setzen, und zwar mit der Funktion

   void change_default_filename (logformat_t *lf, char *newdefault);

Wenn newdefault NULL ist, wird kein Default-Dateiname benutzt.

Zweitens muß man das Flag LF_FLAG_EXPAND_DEFAULT_FILES setzen. (S. im Abschnitt über Flags!)

Man beachte, daß der Default-Filename nicht angehänt wird, wenn die Anfrage des Client nicht mit einem slash endet, weil die Bibliothek dann keine Möglichkeit hat, festzustellen, ob es sich um ein Verzeichnis oder eine Datei handelt. Eigentlich sollten in den Server-Logbüchern Verzeichniszugriffe allerdings immer mit einem slash enden.

Flags

In der Struktur logformat_t gibt es ein Element namens flags, welches eine bitweise Kombination von folgenden Konstanten ist: (vorerst gibt es nur eine)

LF_FLAG_EXPAND_DEFAULT_FILES Ist dieses Flag gesetzt, und ein Default-Dateiname gesetzt, so wird dieser während des Parsens einer Zeile jedem request und jedem referrer, der mit einem slash '/' endet, angehängt.

Accesses

Der Typ access_t

Bevor man eine Zeile parsen kann, muß man erst mal einen Typ haben, in dem man das Ergebnis abspeichern kann. Hier ist er:

   typedef struct {
      unsigned long int requester; /* IP-Adresse des anfragenden Rechners */
                                   /* in networkbyte-order */
      char *command;               /* Kommando (z.B. GET) */
      int rcode;                   /* return-code des Servers (meist 200) */
      int bytes;                   /* Anzahl der zurueckgelieferten Bytes */
      char *request;               /* welche Datei abgerufen wurde */
      char *domain;                /* welche Domain angefordert wurde */
      char *referrer;              /* von wo der surfer kam. (ev. NULL) */
      char *browser;               /* welches Programm angefordert hat */

      /* Datum und Uhrzeit als Zahlen*/
      int dateY, dateM, dateD, timeH, timeM, timeS;
   } access_t;

Die einzelnen Einträge entsprechen den Abkürzungen in der Tabelle im Abschnitt über den Format-String. Die IP-Adresse des Clients liegt allerdings hier als numerische Zahl in networkbyte-order vor. Datums- und Zeitangaben sind ebenfalls stets Zahlen.

Speicherverwaltungsfunktionen

Um mit dem access_t arbeiten zu können, braucht man zunächst einmal einige grundlegende Funktionen:

   access_t *new_empty_access_t ();
   void free_access_t (access_t *a);
   access_t *copy_access_t (access_t *a);

Diese Funktionen sind denke ich selbsterklärend. Die erste Funktion füllt die Variable mit leeren Werten, d.h. NULL-Zeigern für char*-Elemente, 0 für den requester, und -1 für alle anderen numerischen Werte.

Ein- und Ausgabe

Hierzu gibt es nur zwei magere Hilfsfunktionen:

   void fprint_access_t (FILE *f, access_t *a);
   access_t *fscan_access_t (FILE *f);

Die erste Funktion gibt den Inhalt der Variablen a ohne weitere Erklärung aus, aber genau in der Form, in der sie von der zweiten Funktion wieder eingelesen werden kann.

Diese Funktionen sind eher zum Testen und debuggen gedacht und sind mit Vorsicht zu genießen.

Default-Filename hinzufügen

Ist in einem Log-Format das Flag LF_FLAG_EXPAND_DEFAULT_FILES gesetzt, so wird, wie oben erklärt, während des Parsens einer Log-Zeile an alle requests und refferrer, die mit einem slash "/" enden, der default-filename drangehängt, sofern ein solcher existiert. Dies kann man auch manuell machen: Die Funktion

   void add_default_files (access_t *a, logformat_t *lf);

macht genau dies: Sie fügt den Default-Filename des Log-Formates lf (sofern dieser existiert) an den request und den refferrer von a an, sofern diese mit einem slash enden. Dies wird unabhängig davon gemacht, ob das oben erwähnte Flag gesetzt ist oder nicht.

Parsen einer Log-Zeile

Diese zentrale Funktion der Bibliothek sieht wie folgt aus:

   access_t *parse_log_line (char *line, logformat_t *lf);

Es wird versucht, die Zeile line mit Hilfe des Log-Formates lf zu parsen. War dies erfolgreich, so wird das Ergebnis in einem neu allokierten access_t* zurückgegeben. Bei einem Fehler wird NULL zurückgegeben.

Es ist natürlich selbst im Erfolgsfall möglich, oder sogar wahrscheinlich, daß nicht alle Elemente des zurückgegebenen access_t* gesetzt sind. In diesem Fall haben sie immer noch die von new_empty_access_t gesetzten "leeren" Werte.

Hinweise zu eigenen Implementationen

Änderungen des access_t

Änderungen in diesem Datentyp vorzunehmen, ist ganz einfach: Man ändert einfach die Typ-Definition und die Funktionen

   access_t *new_empty_access_t ();
   void free_access_t (access_t *a);
   access_t *copy_access_t (access_t *a);
   int access_t_equal (access_t *a1, access_t *a2);
   void fprint_access_t (FILE *f, access_t *a);
   access_t *fscan_access_t (FILE *f);
   void add_default_files (access_t *a, logformat_t *lf);

Hierbei muß die letzte Funktion wahrscheinlich nur in den seltensten Fällen angefaßt werden.

Hinzunahme oder Entfernen von geparsten Informationen

Hat man den access_t geändert, beispielsweise ein Element hinzugefügt oder eines entfernt, möchte man wohl in der Regel auch, daß diese Informationen aus Log-Dateien herausgezogen werden können. Dies geschieht wie folgt: In der Datei ploglib.c gibt es eine globale Variable namens subterm, die im Augenblick so definiert ist:

   tt_type_t subterm[] = {
     {'S', "([0-9]{1,3}\\.[0-9]{1,3}\\.[0-9]{1,3}\\.[0-9]{1,3})"},  /* numerische IP des Clients */
     {'C', "([a-zA-Z]+)"}, /* Kommando des Clients */
     {'c', "([0-9]+)"},    /* Antwortcode des Servers */
     {'r', "(.*)"},        /* angeforderte Datei */
     {'b', "([0-9\\-]+)"}, /* Anzahl der zurueckgelieferten Bytes */
     {'d', "(.*)"},        /* angeforderte Domain */
     {'f', "(.*)"},        /* referrer */
     {'B', "(.*)"},        /* Browser */

     {'Y', "([0-9]{4})"},    /* 4-ziffriges Jahr */
     {'M', "([0-9]{1,2})"},  /* Monat, 1- oder 2-ziffrig */
     {'N', "([a-zA-Z]{3})"}, /* Monat, 3-buchstabige Abk. */
     {'D', "([0-9]{1,2})"},  /* Tag, 1- oder 2-ziffrig */
     {'E', "([a-zA-Z]{3})"}, /* Tag, 3-buchstabige Abk. */
     {'h', "([0-9]{1,2})"},  /* Stunde, 1- oder 2-ziffrig */
     {'m', "([0-9]{1,2})"},  /* Minute, 1- oder 2-ziffrig */
     {'s', "([0-9]{1,2})"},  /* Sekunde, 1- oder 2-ziffrig */
     {'\0', NULL}
   };

Der Typ tt_type_t besteht aus einem char, und einem String. Bei der Variablen subterm handelt es sich um einen nullterminierten array davon. Die Buchstaben entsprechen den Buchstaben, die man im Format-String angibt. Diese werden bei der Anlage eines neuen Log-Formates durch die entsprechenden regulären Ausdrücke, die dahinter stehen, und die jeweils in runde Klammern eingefaßt sein müssen, da der Substring herausgelöst werden soll, ersetzt. Man kann in obiger Definition beispielsweise sehen, daß %S ersetzt wird durch einen regulären Ausdruck, der auf Zeichenketten der Form x.x.x.x, wobei x aus 1 bis 3 Ziffern besteht, paßt. Will man weitere Informatione parsen, so muß man zunächst einmal diese Variable erweitern.

Anschließend muß man noch den eigentlichen Parser, die Funktion parse_log_line anpassen, die die Verbindung zwischen den Typen logformat_t und access_t darstellt. In dieser befindet sich eine große switch-Anweisung, die die richtigen Informationen in einen neuen access_t schreibt. Sie fängt im Augenblick wie folgt an:

    switch (subterm[lf->series[i-1]].letter) {
    case 'S': /* IP des Clients */
      res->requester=inet_addr(tmpstr);
      free (tmpstr);
      break;
    case 'C': /* Kommando */
      res->command=tmpstr;
      break;
    case 'c': /* Antwortcode */
      res->rcode = atoi(tmpstr);
      free (tmpstr);
      break;
    case 'r': /* angeforderte Datei */
      res->request = tmpstr;
      break;

In der lokalen Variable tmpstr ist der herausgelöste Teil-String der Zeile enthalten, der die Information enthält. Dieser ist neu allokiert. Wird der String selber nicht gebraucht, beispielsweise weil wie oben lediglich eine Integer-Zahl daraus gemacht wird, sollte man ihn wieder freigeben. Die Variabel res (für result) enthält den access_t, der von der Funktion am Ende zurückgegeben wird, und der die geparsten Informationen enthalten soll.

Will man weniger oder mehr Informationen parsen, so muß man diese switch-Anweisung ebenfalls ändern.

Interna des logformat-Typs

Will man noch mehr in den Funktionen rumbasteln, ist es u.U. notwendig, den Typ logformat_t zu ändern:

   typedef struct {
     char *user_format;        /* der string, wie ihn Benutzer angegeben hat*/
     regex_t pre_rex;          /* kompilierter regulaerer Ausdruck */
     int *series;              /* nullterminierter array, der angibt, welche
                                  Bedeutung die einzelnen substrings des 
                                  regulaeren Ausdrucks haben. Bezieht sich auf
                                  die Nummer des Variablen subterm[] */
     size_t nmatch_size;       /* Anzahl der Substrings im regulaeren 
                                  Ausdruck*/
     char mon3[12][4];         /* dreibuchstabige Monatsabk. 0=Januar */
     char day3[ 7][4];         /* dreibuchstabige Tagesabk. 0=Sonntag */

     int flags;
     char *default_filename; 
   } logformat_t;

user_format enthält die Zeichenkette, so wie ihn der Anwender an die Funktion new_logformat übergeben hat, und zwar ohne jegliche Änderungen. Die Funktion nimmt dann die durch die globale Variable subterm gegebenen Ersetzungen vor, übersetzt den entstandenen regulären Ausdruck, und speichert den übersetzten Ausdruck dann im Element pre_rex ab (für "precompiled regular expression").

Weiterhin muß noch festgehalten werden, wie viele herauszulösenden Unterausdrücke der reguläre Ausdruck enthält (es sind nmatch_size) und welche Bedeutung diese haben. Hierzu dient das Element series: Der erste Unterausdruck (hier repräsentiert durch series[0], bezieht sich auf die Information subterm[series[0]]. Wäre im obigen Beispiel also series[4]==1, so wäre der fünfte herausgelöste Unterausdruck das Kommando des Clients, denn subterm[1] steht eben hierfür.

Die Elemente mon3 und day3 enthalten die dreibuchstabigen Monats- und Tagesabkürzungen. Die Elemente default_filename und flags sind glaube ich selbsterklärend.
